# v0.1.4 版本化 SaveGame 与迁移基础

**状态**：Completed

**直接上游**：v0.1.3

**直接下游**：v0.1.5

**未来消费者**：v0.3.0、v0.4.2、v0.4.3、v0.4.4、v0.5.0、v0.5.2、v0.8.2、v0.8.4、v0.9.3

## 目标

- 建立原生、版本化、可迁移的 `USaveGame` 与 `UGameInstanceSubsystem`。
- 使用 A/B 双代保存，确保新写入失败或截断时仍保留上一有效代。
- 使用稳定 `PRSV` 封装、长度与 CRC32，准确区分不存在、空文件、格式错误、损坏、错误 Save 类和未来版本。
- 建立严格递增的 Schema 迁移框架，但不伪造不存在的历史 ProjectR 存档版本。
- 冻结同步加载、异步保存、并发请求合并、回调线程和退出生命周期。
- 为未来 UI、关系、Account、Run、Meta、Settings、Memory 与 Steam Cloud 提供稳定的只读状态和事件面。
- 保证自动化只使用唯一测试槽，不读取、覆盖或删除用户槽。

## 本次范围

- 新增 Save 公共类型、最小 SaveGame 根对象、封装/存储层、迁移器与 Save Subsystem。
- Schema 1 只持久化稳定 `ProfileId`、`SchemaVersion` 与 `SaveRevision`。

## `UPRSaveSubsystem` 公共 API

`UPRSaveSubsystem` 固定继承 `UGameInstanceSubsystem`，公共 C++ API 为：

```text
EPRSaveResult LoadDefaultProfile(FGuid& OutRequestId)
EPRSaveResult CreateNewDefaultProfile(FGuid& OutRequestId)
EPRSaveRequestStatus RequestSaveCurrentProfile(FGuid& OutRequestId)
FPRSaveRuntimeState GetSaveRuntimeState() const
FPRSaveOperationEventNative& OnSaveOperation()
```

- 所有入口仅限 C++，不暴露任意 Blueprint 保存、加载、槽名或删除 API。
- 所有入口必须在 Game Thread 调用；其他线程返回 `InvalidRequest`/`Invalid` 并记录明确日志。
- 所有完成事件在 Game Thread 广播。
- Initialize 只进入 `Ready`，不执行磁盘 I/O。
- Create 仅在没有已加载 Profile 且 A/B 均不存在时成功；任何已占用槽均返回 `SlotOccupied`，不覆盖。
- Create 成功时只生成有效 ProfileId、Schema 1、Revision 0 的内存对象并设置 `bNeedsResave=true`；不在 Create 内写盘，首次落盘必须由显式 Save 请求触发。
- Load 在已有已加载 Profile 时返回 `AlreadyLoaded`，避免覆盖未保存内存状态。
- Save 在没有已加载 Profile 时返回 `RejectedNoProfile`。
- 查询和事件为只读数据面；Save 不 include UI、关系、Account、Run、Meta、Settings、Steam 或未来消费者类型。

## Schema 与迁移合同

- `CurrentSchemaVersion = 1`。
- `MinimumMigratableVersion = 1`。
- Schema 0 返回 `MissingSchemaVersion`。
- 小于最小可迁移版本返回 `UnsupportedOldVersion`。
- 大于当前版本返回 `FutureSchemaVersion`。
- v0.1.4 不注册生产迁移步骤，因为不存在旧 ProjectR Schema。
- 迁移注册表只接受准确 `N -> N+1` 步骤，禁止跳版本。
- 迁移在已反序列化对象的临时副本上顺序执行。
- 每步必须显式成功并把版本恰好增加 1；任一步失败即丢弃副本并返回 `MigrationFailed`。
- 最终 Schema 和数据验证全部成功后才可替换运行时对象。
- 成功迁移只设置 `bNeedsResave=true`，不得在 Load 中自动写回。
- 不新增 Project CustomVersion GUID；业务 Schema 负责 ProjectR 数据迁移，UE SaveGame Header 继续负责引擎序列化版本。

未来持久化分区只按实际版本加入：

- Schema 1：`Profile`，当前只含 ProfileId。
- v0.3.0：关系数据。
- v0.4.3：Account/Run/Graveyard。
- v0.4.4：MetaProgression。
- v0.8.2：Settings。
- v0.5.2：有界 Memory/RunSummary。
- 任何新增分区都必须增加 SchemaVersion 并提供上一版到新版的准确迁移。
- 不创建空的未来占位 Struct 或字段。

## `PRSV` 物理封装

每个物理槽使用固定 16 字节 little-endian Header：

| Offset | 类型 | 值 |
|---|---|---|
| 0-3 | ASCII[4] | `PRSV` |
| 4-5 | `uint16` | `EnvelopeVersion=1` |
| 6-7 | `uint16` | `HeaderSize=16` |
| 8-11 | `uint32` | `PayloadSize` |
| 12-15 | `uint32` | `PayloadCRC32` |
| 16 起 | bytes | `UGameplayStatics::SaveGameToMemory` 生成的 Payload |

规则：

- CRC 使用 `FCrc::MemCrc32`，只覆盖 Payload。
- Payload 必须大于 0 且不超过 16 MiB。
- 文件总长度必须恰好等于 `16 + PayloadSize`。
- 错误 Magic、HeaderSize、长度或 EnvelopeVersion 0 返回 `InvalidEnvelope`。
- EnvelopeVersion 大于 1 返回 `FutureEnvelopeVersion`，不得降级读取或覆盖。
- CRC 不匹配或 Payload 截断返回 `CorruptData`。
- 成功解封后若不是 `UPRSaveGame`，返回 `WrongSaveClass`。
- 本版本不压缩、不加密、不增加校验侧车文件。

- 非 Editor、非 Shipping：`ProjectR_Development_Profile_0`。
- Shipping：`ProjectR_Profile_0`。

- `<Base>_B`

- 生产公共 API 不接收任意槽名。

## A/B 加载规则

- 两代均不存在：返回 `NotFound`，不自动 Create。
- 任一代包含未来 Envelope 或未来 Schema：整体返回对应 Future 结果，不回退到旧代，不覆盖任何文件。
- 只有一代有效、另一代不存在：返回 `Success`。
- 只有一代有效、另一代损坏、不可读或旧版不支持：加载有效代，返回 `RecoveredFromAlternate`，设置 `bNeedsResave=true`。
- 两代有效且 ProfileId 相同：选择最高 SaveRevision。
- 两代 SaveRevision 相同且 Payload 完全相同：确定性选择 A。
- 两代 SaveRevision 相同但 Payload 不同，或 ProfileId 不同：返回 `GenerationConflict`，不加载。
- Load 失败不得替换已发布对象；合同本身禁止在已有对象时再次 Load。

两代均无有效数据时，公开错误优先级固定为：

1. Future envelope/schema。
2. Unsupported old version。
3. Migration failure。
4. Missing schema。
5. Wrong Save class。
6. Corrupt data。
7. Invalid envelope。
8. Empty data。
9. Read failure。
10. Not found。

日志必须同时记录 A/B 各自的独立诊断，不能只输出最终聚合结果。

## 保存、并发与退出生命周期

- Load 同步执行；16 MiB 上限和仅启动/菜单调用构成固定边界。
- Save 在 Game Thread 创建不可变快照和封装，再通过平台 SaveGame System 异步写入。
- 同一 Subsystem 同时只允许一个物理写操作，不能依赖平台全局异步队列的未保证顺序。
- 保存开始前重新探测 A/B：发现未来版本、不同 Profile、意外更高 Revision 或外部冲突时返回 `GenerationConflict`/Future 结果，不写入。
- 新 Profile 首次写 A；已加载 A 时写 B，已加载 B 时写 A；两代有效时只覆盖较旧、非当前有效代。
- 从损坏备用代恢复后，只有显式 Save 请求可以修复该备用代；Load 不自动修复。
- 永不覆盖包含未来 Envelope 或未来 Schema 的代。
- 新快照 SaveRevision 为当前已发布 Revision+1；溢出返回 `InvalidRequest`。
- 写入成功后必须重新读取并完整验证目标代的 Envelope、CRC、Save 类、ProfileId、Schema 和 Revision。
- 只有写后验证成功，快照和新 Revision 才成为当前运行时状态。
- 写入或验证失败时保留上一有效代和当前运行时对象。
- 保存进行中收到第一次重复请求时，建立一个 trailing 请求并返回 `Coalesced`；trailing 快照采用 last-write-wins。
- 后续重复请求替换 trailing 快照并返回同一 trailing RequestId，不扩大队列。
- 当前请求成功后，trailing 请求以新 Revision 为基线启动。
- 当前请求失败时，trailing 请求返回 `CancelledAfterPriorFailure` 并清空。
- Loading/Saving 状态不接受其他类型操作；返回 `Busy`/`RejectedBusy`。
- Deinitialize 切换为 `ShuttingDown`、拒绝新请求，并为未启动 trailing 请求广播 `CancelledOnShutdown`。
- 退出时不执行同步兜底保存、不等待磁盘、不阻塞进程。
- in-flight 回调只能通过弱 Subsystem 引用返回；Subsystem 已销毁时不得访问运行时对象或广播。

## 自动测试槽规则

- 测试逻辑槽格式固定为 `ProjectR_Automation_<32位小写十六进制GUID>`。

## 回滚和停止条件

出现以下任一情况必须停止：

- 必须修改 Core GameInstance、Ability、Combat、Input、Config 或 UE Package 才能继续。
- 不能通过 `ISaveGameSystem` 保持平台无关存储。
- 任一写入可能覆盖当前有效代、未来版本代或真实用户槽。
- 写失败或验证失败后上一有效代不可加载。
- 未来版本被静默降级读取或覆盖。
- 并发请求产生两个物理写入、乱序 Revision 或重复完成事件。
- 自动化访问非当前测试 GUID 的槽，或准确清理失败。
- 任一历史回归、Editor 新鲜加载、PIE 或 Dirty=0 失败。

回滚规则：

- C++、Build.cs 和文档只反向撤销本版本准确 diff。
- 自动化只删除当前测试 GUID 的准确 `_A`、`_B`。
- 不通过 Shell、普通文件 I/O、通配符或递归操作 Save 文件。
- 不删除、重命名或覆盖真实用户槽；发现真实槽时隔离并报告。
- 不回滚 v0.1.0-v0.1.3 的 Source、Config、Package、Tag 或自动化工具。
- 禁止硬重置、Git clean、Save All、批量删除和 Fix Redirectors。
