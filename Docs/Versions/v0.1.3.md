# v0.1.3 技能授予、冷却、资源消耗和输入 Tag

**状态**：Completed

**直接上游**：v0.1.2

**直接下游**：v0.1.4

**主要消费者**：v0.1.5、v0.2.0、v0.2.3、v0.3.2、v0.4.2、v0.4.4、v0.8.2

## 目标

- 在冻结的 `UPRAbilitySystemComponent` 上增量建立通用 Ability 基础设施。
- 建立数据驱动 AbilitySet、Authority 授予/移除、幂等记录和准确回滚。
- 将既有 InputTag Pressed/Released Hook 唯一映射到 AbilitySpec，并在 ASC 内维护 Held 状态。
- 冻结 `OnInputTriggered`、`WhileInputActive`、`Passive` 三种激活策略。
- 通过标准 GAS GameplayEffect 建立 Energy Cost、Cooldown、失败原因和 Commit。
- 提供稳定原生生命周期事件和只读查询面，供未来 HUD、QTE、构筑和统计消费。
- 保持 PlayerState-owned Spec 跨 Pawn 替换、死亡和复活的正确生命周期。

- v0.1.2 的 ASC、AttributeSet、CombatSubsystem、DamageRequest、CombatEvent、死亡和复活状态机继续作为上游事实源；Ability 只响应其状态，不复制其职责。

- v0.1.5 未来只能通过受控 ASC API 查询或授予，不在本版本提前创建 Debug 命令或菜单。
- v0.2.0 的六个正式技能继承 `UPRGameplayAbility`、填充正式 AbilitySet，并通过 `FPRDamageRequest` 提交伤害。
- v0.2.3 HUD 只订阅生命周期事件和只读查询面；ASC 不依赖 UI 类型。
- v0.3.2 QTE 可订阅 Ability 生命周期与 CombatEvent，但不得反向调用 Ability 私有实现。
- v0.4.2 与 v0.4.4 的奖励和构筑通过 AbilitySet Handle 幂等授予/移除，不复制授予记录。
- v0.8.2 重绑定只改变 InputTag 到 InputAction/Key 的输入配置；AbilitySpec 中稳定 InputTag 不变，不重新授予 Ability。
- 本版本不新增 Save 字段、正式技能内容、未来消费者类引用或 network/LLM 等待路径。

## `UPRGameplayAbility` 合同

- 基类固定为 `UGameplayAbility` 的 `InstancedPerActor` 扩展。
- 冻结以下 CDO 配置：
  - `FGameplayTag AbilityTag`。
  - `EPRAbilityActivationPolicy ActivationPolicy`。
  - `bool bCanActivateWhileDead = false`。
  - `bool bCancelOnDeath = true`。
- InputTag 不作为 Ability CDO 属性保存，只能从当前 AbilitySpec 解析。
- AbilityTag 在授予时加入 Spec 的动态源 Tag。
- Cost 和 Cooldown 的唯一权威来源分别为继承的 `CostGameplayEffectClass` 与 `CooldownGameplayEffectClass`。
- Ability 实现必须在产生伤害、位移或其他业务后果前成功调用 `CommitAbility`。
- 基类不自动产生伤害或 CombatEvent；v0.2.0 的正式 Ability 使用自己的 AbilityTag 构造 `FPRDamageRequest`。
- Dead 且 `bCanActivateWhileDead=false` 时，失败 Tag 至少包含 `Ability.ActivateFail.TagsBlocked` 和 `State.Dead`。

## AbilitySet 验证和授予规则

- `UPRAbilitySetDataAsset` 继承 `UPrimaryDataAsset`，提供稳定 PrimaryAssetId。
- `AbilityLevel` 必须大于等于 1。
- AbilityClass 必须继承 `UPRGameplayAbility`，且 AbilityTag 有效。
- `OnInputTriggered` 和 `WhileInputActive` 必须配置一个 `Input.*` Tag。
- `Passive` 必须不配置 InputTag。
- `GrantedSpecTags` 不得包含任何 `Input.*` Tag；InputTag 只能由专用字段提供。
- 同一个 AbilitySet 内不得重复 AbilityTag 或 InputTag。
- 同一个 ASC 同时最多有一个 Spec 使用同一 AbilityTag、同一 InputTag。
- Spec 的数值 `InputID` 固定为 `-1`，不得建立第二套数值输入映射。
- Spec 的 `GetDynamicSpecSourceTags()` 固定包含：
  - Ability CDO 的 AbilityTag。
  - Entry 的唯一 InputTag，Passive 除外。
  - Entry 的 GrantedSpecTags。
- Spec SourceObject 为 `AbilityData`；为空时使用 AbilitySet 自身。
- 授予只允许 Authority：
  - 客户端调用返回 `NotAuthoritative`，不改变状态。
  - 首次成功返回 `Applied` 和有效 GrantId。
  - 对同一 AbilitySet、同一模式重复调用返回 `AlreadyApplied` 和原 Handle。
  - `InitializationOnly` 只授予 `bGrantOnInitialization=true` 的 Entry。
  - 后续 `AllEntries` 只补充此前未授予的 Entry，并保持同一 GrantId。
- 任何结构错误或 `GiveAbility` 失败时，撤销本次调用中新创建的全部 Spec，保留调用前状态。
- `RemoveAbilitySet` 只按内部 GrantId 移除准确 Spec；活跃 Ability 先取消，Held 状态清除，然后移除。
- 对已移除或伪造 Handle 再次移除返回 `NotFound`，不得影响其他 Ability。

- 授予、移除和输入方法是受控 C++ API，不开放任意 Blueprint 授予或移除入口。
- 查询和事件是只读数据面；Combat、UI、QTE 或 Save 类型不得进入 ASC。
- 生命周期事件是 ASC 本地上下文事件，不作为第二套 CombatEvent，也不自行跨网络复制。

## InputTag 唯一路由

- `APRPlayerCharacter::HandleInputTagPressed` 只调用当前 ASC 的 `AbilityInputTagPressed`。
- `APRPlayerCharacter::HandleInputTagReleased` 只调用当前 ASC 的 `AbilityInputTagReleased`。
- ASC 只对 `GetDynamicSpecSourceTags()` 中精确匹配该 InputTag 的唯一 Spec 操作。
- Press：
  - 第一次 Press 设置 `Spec.InputPressed=true`，加入 Held Handle 集合，并调用 GAS `AbilitySpecInputPressed`。
  - 同一物理按住期间重复 Press 不再次激活、不重复事件。
- Release：
  - 仅对已经 Held 的 Handle 生效。
  - 先调用 `AbilitySpecInputReleased`，再移除 Held 状态。
  - 重复 Release 无操作。
- 不修改 Enhanced Input Trigger、InputAction、IMC 或 Mapping Context。

## 激活策略

### `OnInputTriggered`

- 只在 Press 上升沿尝试激活一次。
- 若已 Active，不重触发。
- Release 只转发给 Active Ability，不自动结束或取消。
- Cooldown、Cost、Dead 等失败只产生一次 ActivationFailed。

### `WhileInputActive`

- Press 上升沿尝试激活一次并保持 Held。
- 激活失败后不在按住期间轮询重试；必须 Release 后再次 Press。
- Release 转发输入后取消当前 Ability，产生一次 Cancelled。
- 重复 Press/Release 不重复激活或取消。

### `Passive`

- 不接受 InputTag。
- 成功授予且 ActorInfo/Avatar 可用、目标允许激活时自动尝试一次。
- Dead 且不允许死亡激活时保留 Spec，等待 Revive，不回滚授予。
- Avatar 更新或 Revive 后，只对 inactive Passive 尝试恢复一次。
- 移除 AbilitySet、死亡或 Avatar 清理时按 `bCancelOnDeath` 和生命周期规则取消。

## Cooldown 与 Energy Cost

- Cost GE 必须：
  - Instant。
  - 恰好一个对 `UPRAttributeSet::Energy` 的 Additive 负 Modifier。
  - 无 Execution、Period、Granted Ability、GameplayCue 或 Stacking。
- Cooldown GE 必须：
  - Has Duration，Duration 大于 0。
  - 无属性 Modifier 或 Execution。
  - 恰好授予一个 `Cooldown.*` Tag。
  - 同一 ASC 的不同 Ability 默认不得共享 Cooldown Tag。
- UE 5.8 Commit 顺序冻结为：
  1. `CheckCooldown`。
  2. `CheckCost`。
  3. `ApplyCooldown`。
  4. `ApplyCost`。
  5. Notify Commit。
- 前两个检查不得产生状态变化。
- `UPRGameplayAbility::CommitAbility` 保留上述顺序并验证实际应用结果：
  - Cooldown 应用失败时不应用 Cost。
  - Cost 在检查后异常应用失败时，只移除本次 Commit 创建的 Cooldown Handle。
  - 回滚失败是 required FAIL，必须停止版本。
  - Instant Cost 成功以 `WasSuccessfullyApplied()` 判断，不使用 `IsValid()`。
  - 只有 Cooldown 与 Cost 均成功才广播 Committed。
- Cooldown 阻止激活时 Energy 不变；Cost 不足时不创建 Cooldown。
- Query/Event 中的 EnergyCost、CooldownDuration、CooldownRemaining 从相同 GE 配置计算，不建立 UI 专用副本。

## 生命周期事件顺序

- 授予：`Granted`；Passive 可随后 `Activated`。
- 成功触发型：`Activated → Committed → Ended`。
- 激活前失败：仅 `ActivationFailed`。
- Commit 后半段失败：`Activated → CommitFailed → Cancelled`。
- Held Release、死亡或移除导致取消：`Cancelled`。
- 移除活跃 Spec：`Cancelled → Removed`。
- 重复 Press、Release、授予或移除不得制造重复生命周期事件。

## 死亡、复活和 Pawn 生命周期

- ASC 精确监听既有 `State.Dead`，不复制 v0.1.2 的死亡逻辑。
- Dead 进入时：
  - 清除 Held 输入。
  - 取消所有 `bCancelOnDeath=true` 的 Active Ability。
  - 后续 Press 对匹配 Spec 产生带 `State.Dead` 的 ActivationFailed。
- Revive 只由 `UPRCombatSubsystem` 改变生命状态；Dead 移除后输入资格自动恢复。
- Revive 不重新授予 AbilitySet，不改变 Spec Handle。
- Passive 在 Revive 后按合同恢复一次。
- PlayerState 持有 AbilitySpec 和授予记录；Pawn 替换只更新 Avatar。
- Avatar 切换前清除旧 Pawn 的 Held 输入并取消依赖旧 Avatar 的 Active Ability。
- ActorInfo 更新后 Spec Handle 不变，Passive 只对新 Avatar 恢复一次。
- 旧 Pawn 的 Release、EndPlay 或 Timer 不得改变新 Avatar 的 ActorInfo、Held 状态或 Ability。
- 默认属性和初始 AbilitySet 均不得因重新占有重复应用。

## MCP Operation Manifest

本任务合同转换不执行以下操作。v0.1.3 实施时必须在一个 UnrealEditor 实例中串行执行：

1. 确认唯一 Editor、`PIE=false`、初始 Dirty=0，检查七个新路径不存在且可编辑。
2. 重新枚举官方 GASToolsets、BlueprintTools、DataAsset/ObjectTools、ProgrammaticToolset 能力并保存能力审计证据。
3. Build 完成并加载新 C++ 类后，创建两个 GE、三个 GA 和两个 AbilitySet。
4. 在单一 Programmatic 事务中配置两个 GE；逐项断言 Duration、Modifier、Attribute、Magnitude、Granted Tag、Execution、Period、Cue 和 Stacking。
5. 配置三个 GA 的 CDO、父类、AbilityTag、ActivationPolicy、Cost/Cooldown 引用和最小验证 Graph。
6. 配置验证 AbilitySet 的三个 Entry；配置正式空 AbilitySet。
7. 设置 `BP_PlayerState.InitialAbilitySets`。
8. 任一 Schema、类、数组长度、Tag 或引用不一致时抛错、回滚事务并停止，不保存。
9. warnings-as-errors 编译全部 Manifest Blueprint/GE，并验证两个 DataAsset。
10. 只精确保存非空 Manifest，随后检查全部意外 Dirty=0。
11. 正常关闭和重启 Editor，回读所有 CDO、Graph、GE、DataAsset、BP 引用及冻结的 Input/ASC/Combat 资产。
12. 打开 `L_CombatGym`，运行 Ability、Input 和 Combat 固定 PIE 冒烟。
13. StopPIE 后复查日志、截图、ActiveEffect、Spec、Tag、引用和 Dirty=0。
14. 生成 AutomationReport。

当前静态工具清单没有在活动 PIE 中授予验证 AbilitySet 并捕获生命周期事件的入口。实施时若重新审计仍确认该缺口，则创建最小 `RunPIEAbilitySmoke`：

- 只加载固定验证资产。
- 只执行固定授予、输入、Cost、Cooldown、死亡、复活、移除和状态回读。
- 不保存 Package，不执行任意代码，不修改正式 AbilitySet。
- 保持既有 Input/Combat Tool 不变。
- 若官方工具已补足能力，则不创建扩展，optional check 记 `NOT_RUN`。
