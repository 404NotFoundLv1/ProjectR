# v0.1.2 统一伤害、死亡、复活和受击表现

**状态**：In Progress

**直接上游**：v0.1.1

**直接下游**：v0.1.3

## 目标

- 在 v0.1.1 的同一 ASC/AttributeSet 上建立唯一、确定性的 GAS 伤害入口。
- 实现 Shield 优先吸收、spill-over 到 Health、Health Clamp 后死亡、幂等死亡和显式复活。
- 通过稳定原生 CombatEvent 向 HUD、QTE、画像、对话和统计单向发布战斗事实。
- 为当前 Player Pawn 提供不依赖正式动画、VFX 或 SFX 的短硬直、死亡锁定和复活恢复占位反馈。
- 保持 Pawn 替换、重新占有、默认属性一次应用和 v0.1.0 输入移动合同。

## 冻结合同和未来扩展点

- ASC 具体类型继续冻结为 `UPRAbilitySystemComponent`；本版本不得修改或替换该类。
- ASC Owner 永远为 `APRPlayerState`，Avatar 为当前 `APRPlayerCharacter`；Pawn 替换只更新 Avatar。
- AttributeSet 继续为 `UPRAttributeSet`；既有 11 项 replicated 属性名称、类型、范围和 RepNotify 合同不变。
- `IncomingDamage` 是非复制、非持久、无 RepNotify 的 Meta 属性，不计入冻结的 11 项用户属性 Schema。
- CombatEvent 是 Combat 向 HUD、QTE、画像、对话、统计、Boss 和自动化测试发布信息的唯一通用事件面。
- Combat 不得 include 或保存具体 HUD、QTE、画像、对话、统计、技能、敌人或章节类型。
- v0.1.3 只能在同一薄 ASC 上增量实现 AbilitySet/InputTag/Cooldown/Cost，不能重做伤害入口。
- v0.2.0 的技能只能构造 `FPRDamageRequest`，不能各自复制 Shield/Health/Death 逻辑。
- v0.2.1 的敌人 Owner 通过实现 `IPRCombatantInterface` 接入同一入口；Avatar 通过 `IPRCombatFeedbackInterface` 接入占位或正式反馈。
- v0.2.3 和 v0.3.2 只能订阅公共属性事件和 CombatEvent，不反向调用 Combat 私有实现。
- CombatEvent 新类型通过新的 `Combat.Event.*` Tag 增量扩展；不得静默改名现有字段或 Tag。
- 本版本不冻结 Blueprint callable 玩法 API；未来 UI 使用 C++ adapter 订阅原生事件。

## 公共 C++ 接口

### `FPRDamageRequest`

字段顺序和语义冻结为：

```text
FName SourceId
TWeakObjectPtr<UObject> DamageSource
TWeakObjectPtr<AActor> Instigator
TWeakObjectPtr<AActor> Target
FGameplayTag AbilityTag
float RawDamage
bool bCritical
FGameplayTagContainer DamageTags
```

- `SourceId` 是必填稳定来源 ID，不得为 `NAME_None`。
- `DamageSource` 可为空，表示没有 UObject 来源的环境伤害。
- `Instigator` 可为空，表示环境或系统伤害。
- `Target` 必填，且必须是目标 ASC 当前 Avatar；旧 Pawn 或仅传 PlayerState 均为无效请求。
- `AbilityTag` 可为空；v0.1.2 不产生正式 AbilityTag，v0.2.0 可在未来填入稳定 `Skill.*` Tag。
- `RawDamage` 是进入验证和上限 Clamp 前的调用方数值。
- `bCritical` 只传播，不参与 v0.1.2 数值计算。
- `DamageTags` 只承载调用方已确定的通用伤害语义；本版本不创建 DamageType 内容 Tag。

### `FPRCombatEvent`

字段顺序和语义冻结为：

```text
FGuid EventId
FGameplayTag EventTag
FName SourceId
FName TargetId
TWeakObjectPtr<UObject> DamageSource
TWeakObjectPtr<AActor> Instigator
TWeakObjectPtr<AActor> Target
FGameplayTag AbilityTag
float RawDamage
float ShieldAbsorbed
float HealthDamage
float RemainingHealth
float RemainingShield
bool bCritical
FGameplayTagContainer DamageTags
FGameplayTagContainer ResponseTags
bool bFatal
double WorldTimeSeconds
```

- 每次广播创建新的 `EventId`；同一次致死伤害的 Damage 和 Death 事件使用不同 ID。
- `EventTag` 只能使用已注册且由 Combat 所有的事件 Tag。
- `TargetId` 由 ASC Owner 的 `IPRCombatantInterface` 提供，不从 UObject 名称猜测。
- `RawDamage` 是通过合法性验证并 Clamp 后实际提交给 GE 的数值。
- `ShieldAbsorbed`、`HealthDamage`、Remaining 值均来自应用后的实际属性差值，不从理论值冒充。
- `ResponseTags` 只由 Combat 根据实际结果生成。
- `WorldTimeSeconds` 使用当前 World 时间，仅用于同一 World 内排序，不作为跨存档时间。
- 事件中的弱引用仅供当前运行时即时消费；长期消费者保存稳定 ID、Tag 和数值，不持久化 UObject。

### `FPRReviveRequest`

字段顺序和语义冻结为：

```text
FName SourceId
TWeakObjectPtr<UObject> DamageSource
TWeakObjectPtr<AActor> Instigator
TWeakObjectPtr<AActor> Target
float HealthFraction = 1.0f
float ShieldFraction = 1.0f
```

- `SourceId` 和 `Target` 必填。
- `HealthFraction` 必须为有限值且位于 `(0, 1]`。
- `ShieldFraction` 必须为有限值且位于 `[0, 1]`。
- 本版本不允许复活请求改变 Max 属性、位置、Pawn Class 或重生点。

### `EPRCombatRequestStatus`

```text
Applied
RejectedInvulnerable
RejectedDead
RejectedAlive
Invalid
```

- `Invalid` 表示结构、数值、Authority、ASC、Avatar、Combatant 或 GE 前置不成立；不广播 CombatEvent。
- `RejectedInvulnerable` 和 `RejectedDead` 是合法命中尝试的确定性结果，会广播 `Combat.Event.DamageRejected`。
- `RejectedAlive` 只用于重复 Revive，不改状态、不广播事件。

### `UPRCombatSubsystem`

```text
UPRCombatSubsystem : UWorldSubsystem

EPRCombatRequestStatus ApplyDamage(const FPRDamageRequest& Request)
EPRCombatRequestStatus Revive(const FPRReviveRequest& Request)
FPRCombatEventNative& OnCombatEvent()
```

- 只提供 C++ 原生接口，不在本版本暴露 Blueprint callable 玩法 API。
- 负责唯一公共入口、输入校验、ASC/Combatant 解析、EffectContext、SetByCaller、状态机、事件和反馈。
- `FPRCombatEventNative` 是携带 `const FPRCombatEvent&` 的 native multicast delegate。
- Subsystem 不保存具体消费者引用，也不等待网络或 LLM。

### `IPRCombatantInterface`

```text
FName GetCombatantId() const
TSubclassOf<UGameplayEffect> GetDamageEffectClass() const
```

- `APRPlayerState` 首次实现，CombatantId 固定为 `Player`。
- 目标 ASC 继续通过 `IAbilitySystemInterface` 获取；接口不复制 ASC 所有权 API。
- `DamageEffectClass` 由 Owner 的正式 Blueprint 配置，调用方不得选择任意 GE 绕过合同。
- v0.2.1 敌人 Owner 实现相同接口即可接入，不要求 CombatSubsystem include 敌人类型。

### `IPRCombatFeedbackInterface`

```text
void HandleCombatHitReaction()
void HandleCombatLifeStateChanged(bool bIsDead)
```

- 当前由 `APRPlayerCharacter` 实现。
- 接口不接收 HUD、QTE、技能、敌人或 Combat 私有类型。
- 非致死实际伤害调用一次 HitReaction；致死和复活分别调用 LifeStateChanged。

## 伤害输入规则

- 只允许目标 Authority 应用伤害；非 Authority 返回 `Invalid`，不创建 GE 或事件。
- `SourceId` 必须非 `NAME_None`。
- Target 必须有效、实现 `IAbilitySystemInterface`、返回有效 `UPRAbilitySystemComponent`，且等于该 ASC 当前 Avatar。
- ASC Owner 必须实现 `IPRCombatantInterface`，返回非空 TargetId 和有效 `GE_Damage` Class。
- DamageSource 和 Instigator 可以为空，以支持环境伤害。
- `NaN`、正无穷、负无穷、负数和零伤害全部返回 `Invalid`：不应用 GE、不广播 CombatEvent，记录带 SourceId/Target 的明确日志。
- 最大合法归一化 RawDamage 为 `200000.0f`，等于合法 MaxHealth 和 MaxShield 上限之和。
- 大于 `200000.0f` 的有限数值 Clamp 到 `200000.0f` 并记录 Warning；事件中的 RawDamage 为 Clamp 后值。
- 校验顺序固定为：结构/数值/Authority -> Dead -> Invulnerable -> GAS 应用。
- Target 同时具有 Dead 和 Invulnerable 时，Dead 优先，返回 `RejectedDead`。
- Dead 目标广播一次 `Combat.Event.DamageRejected`，ResponseTags 含 `State.Dead`；不产生第二次 Death。
- Invulnerable 目标广播一次 `Combat.Event.DamageRejected`，ResponseTags 含 `State.Invulnerable`；属性不变，不调用 HitReaction。
- Critical 只原样进入事件，不修改 RawDamage、Shield 或 Health。
- DamageTags 原样传播；ResponseTags 只能由 Combat 根据实际结果生成。
- Health 为零却没有 Dead Tag 属于不允许的外部绕过状态：记录 Error 并返回 `Invalid`，不为未知来源伪造 Death 事件。

## Shield、Health 和事件规则

```text
ShieldAbsorbed = Min(BeforeShield, RawDamage)
HealthDamage = Min(BeforeHealth, RawDamage - ShieldAbsorbed)
```

- RemainingShield 和 RemainingHealth 是 GAS 应用并经过 AttributeSet Clamp 后的实际当前值。
- ShieldAbsorbed 大于零时增加 `Combat.Response.ShieldAbsorbed`。
- BeforeShield 大于零且 RemainingShield 为零时增加 `Combat.Response.ShieldBroken`，包括恰好打空但未伤 Health 的情况。
- HealthDamage 大于零时增加 `Combat.Response.HealthDamaged`。
- 成功非致死伤害只广播一个 `Combat.Event.Damage`。
- 致死伤害先广播 `Combat.Event.Damage`（`bFatal=true`），再进入 Dead 并广播一个独立 EventId 的 `Combat.Event.Death`。
- ShieldBreak 不另发第二个事件；未来 HUD/QTE 从 Damage 事件 ResponseTags 读取，避免重复统计。
- 非致死且实际吸收或扣血大于零时调用一次 HitReaction。
- 致死时只调用 Death feedback，不叠加短硬直。
- 非法请求不广播事件；Invulnerable/Dead 属于合法命中尝试，使用 DamageRejected 事件。
- DamageRejected 的 ShieldAbsorbed 和 HealthDamage 为 0，Remaining 值为拒绝时的实际属性，`bFatal=false`。
- Death 事件复用导致死亡的 Damage 字段，并在 ResponseTags 增加 `State.Dead`。
- Revive 事件的伤害数值字段为 0，Remaining 值为复活后的实际属性，ResponseTags 含 `State.Alive`。

## 死亡状态机、复活和 Pawn 生命周期

### 状态机

```text
Alive --fatal damage--> Dead
Dead --valid revive--> Alive
Dead --damage--> Dead (rejected, no duplicate Death)
Alive --revive--> Alive (RejectedAlive, no event)
```

- Authority 上初始化成功且 Health 大于零时，ASC 保持一个 replicated loose `State.Alive`。
- 初始化时若已经具有 `State.Dead`，重新占有不得添加 Alive 或复活。
- 致死时原子地移除 Alive、添加 replicated loose Dead，并保证 Death 事件只广播一次。
- Alive 和 Dead 不得同时存在；自动化必须验证所有过渡。
- `State.Invulnerable` 只作为伤害入口的读取条件；本版本不创建正式无敌技能或 GameplayEffect。

### Pawn 和 PlayerState

- 死亡状态、ASC 和属性归 PlayerState 所有，跨 Pawn 替换保留。
- Dead PlayerState 占有新 Pawn 时只更新 ASC Avatar；不重放默认 GE，不改变 Health/Shield，不自动复活。
- 新 Character 在 GAS 初始化完成后立即读取 Dead Tag 并同步死亡占位表现。
- 旧 Pawn 不能清除或改变新 Avatar 的 ActorInfo、死亡表现或复活目标。
- Alive 状态下 Pawn 替换继续继承当前 Health、Shield、MoveSpeed 和其他 v0.1.1 属性。

### Revive

- `Revive` 只接受 Dead 且等于 ASC 当前 Avatar 的 Target。
- 使用 ASC 数值 API 将 Health 设置为 `MaxHealth * HealthFraction`，Shield 设置为 `MaxShield * ShieldFraction`。
- 属性设置完成并验证 Health 大于零后，移除 Dead、添加 Alive、广播 Revive，再恢复当前 Avatar 表现。
- 任一属性设置或状态验证失败时返回 `Invalid`，不得只完成一半状态过渡。
- 对 Alive 目标重复 Revive 返回 `RejectedAlive`，不改属性、不广播事件。
- v0.1.2 不销毁或 Spawn Pawn；未来系统可先替换 Pawn，再对当前 Avatar 调用相同 Revive API。
- 复活不重放 `GE_DefaultAttributes`，不改变默认属性一次性状态。

### 当前占位反馈

- 死亡表现只禁用 CharacterMovement，保留 Actor、碰撞、相机、Controller 和 PlayerState。
- HitReaction 默认硬直 `0.10s`，使用 Timer 暂停并恢复受击前 MovementMode，不使用 Tick。
- 重复非致死命中重新启动同一个 Timer，不叠加多个恢复回调。
- 致死时清除 HitReaction Timer，并保持 MovementMode=None。
- 复活统一恢复 Walking。
- 不调用正式动画、Montage、VFX、SFX、Camera Shake、震动、HUD 或 DamageNumber。

## GAS 职责边界

### `UPRCombatSubsystem`

- 进行唯一公共请求验证。
- 从 Instigator 的 ASC 创建 Source Spec；Instigator 无 ASC 时使用 Target ASC 作为确定性环境伤害 Source ASC。
- DamageEffect Class 始终来自 Target ASC Owner 的 `IPRCombatantInterface`。
- 在 EffectContext 中写入 Instigator 和 DamageSource，在 Spec 中设置 `Combat.Data.Damage`。
- 使用 `WasSuccessfullyApplied()` 判断 Instant GE 是否成功，不使用 `IsValid()`。
- 在同步 GE 应用前后读取 Shield/Health，生成实际 CombatEvent、状态过渡和反馈。

### `/Game/ProjectR/Effects/GE_Damage`

- Parent：`/Script/GameplayAbilities.GameplayEffect`。
- DurationPolicy：Instant。
- Modifiers：0。
- Executions：恰好 1 个 `UPRDamageExecutionCalculation`。
- 不配置 Period、Granted Tags、Application Requirements、Stacking、GameplayCue、Granted Ability 或额外 GE Component。
- GE 不直接修改 Shield 或 Health，避免和 AttributeSet 重复扣血。

### `UPRDamageExecutionCalculation`

- 只读取 SetByCaller `Combat.Data.Damage`。
- 只接受有限且位于 `(0, 200000]` 的值；直接绕过 Subsystem 的非法 Spec 输出 0 并记录明确日志。
- 唯一输出是对 `UPRAttributeSet::IncomingDamage` 的 Additive Modifier。
- 不计算 Critical、AttackPower、抗性、技能公式、死亡、复活、事件或反馈。

### `UPRAttributeSet::PostGameplayEffectExecute`

- 识别 `IncomingDamage` 后立即保存合法正值并将 Meta 属性清零。
- 按 Shield 优先、剩余进入 Health 的规则精确修改既有属性。
- 继续使用既有 Clamp 保证 Shield/Health 合法。
- 不发布 CombatEvent、不切换死亡状态、不调用反馈、不重复扣血。
- 其他既有 11 项属性的 Clamp、Max 比例和 RepNotify 行为保持不变。

### 普通 C++ 数值调用

- 禁止作为伤害入口；业务调用方不得直接设置 Shield/Health 表示伤害。
- 只允许默认初始化、原生自动化测试准备和受控 Revive 使用 ASC 数值 API。
- 测试直接赋值不得进入 Shipping 玩法接口。

## GameplayTag 增量

本版本需要新增以下八个显式 Tag，预期显式总数从 53 变为 61，根命名空间从 9 变为 10：

| Tag | 所有者/消费者 | 兼容影响与理由 |
|---|---|---|
| `Combat.Data.Damage` | CombatSubsystem、GE Execution | 唯一 SetByCaller 数值键；避免调用方自选字段 |
| `Combat.Event.Damage` | HUD、QTE、画像、对话、统计 | 成功伤害的稳定事件类型 |
| `Combat.Event.DamageRejected` | 受击反馈、统计 | 区分 Dead/Invulnerable 拒绝与成功伤害 |
| `Combat.Event.Death` | HUD、QTE、对话、统计 | 幂等死亡事件 |
| `Combat.Event.Revive` | HUD、QTE、统计 | 显式复活事件 |
| `Combat.Response.HealthDamaged` | HUD、QTE | 标识 Health 实际减少 |
| `Combat.Response.ShieldAbsorbed` | HUD、QTE | 标识 Shield 实际吸收 |
| `Combat.Response.ShieldBroken` | HUD、QTE | 标识 Shield 从正数变为零 |

- `State.Alive`、`State.Dead`、`State.Invulnerable` 继续复用，不重命名。
- `UPRTagLibrary` 增加上述八个 checked getter，总数从 16 变为 24。
- 不添加临时技能、敌人、QTE、HUD、Critical、DamageType、Heal 或 Revive Source Tag。
- 新 Combat 根、事件字段、状态过渡和事件演进规则必须在 v0.1.2 ADR 中记录。
- 发布后只允许增加有所有者和消费者的子 Tag；重命名或删除需要 Redirect、ADR 和兼容测试。
