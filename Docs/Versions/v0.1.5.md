# v0.1.5 隔离式 Debug 菜单、结构化开发命令与日志分类

**状态**：Completed

**直接上游**：v0.1.4

**直接下游**：v0.2.0

**未来消费者**：v0.2.0、v0.2.1、v0.2.2、v0.3.1、v0.3.2、v0.4.0、v0.4.3、v0.9.1、v1.0.1、v1.0.2

## 目标

- 建立可扩展、结构化、白名单式的 Debug 命令注册与执行框架。
- 在 Development Editor 和 Development Client 中提供数据驱动的原生 Slate 面板。
- Debug 只调用现有受控 Gameplay、Save 和 Travel API，不形成第二套运行时实现。
- 将 Debug 插件从 Shipping 构建、收据、模块和运行时行为中整体排除。
- 冻结 Core、Input、GAS、Ability、Combat、Save、Debug 日志类别和敏感数据脱敏合同。
- 保持 v0.1.0-v0.1.4 的 Input、GAS、Combat、Ability、Save、地图、GameplayTag 和 UE Package 合同不变。

- v0.1.4 正式提交为 `d746de9e56ab8ec8af657c01b9100315fbff45be`（`v0.1.4 Add versioned SaveGame subsystem`），且已与 `origin/main` 同步。
- v0.1.4 的 `ProjectR.Save` 5/5、`ProjectR.Ability` 5/5、`ProjectR.GAS` 4/4、`ProjectR.Combat` 4/4、`ProjectR.Input` 3/3 均为 PASS；AutomationReport 为 34/34 required PASS。

- `UPRCombatSubsystem` 已提供唯一 `ApplyDamage`/`Revive` 入口；`UPRSaveSubsystem` 已提供只读 `GetSaveRuntimeState`；`UPRGameInstance` 已提供固定 `EPRMapId` 的 `OpenMap`。
- `UPRAbilitySystemComponent` 已提供正式 AbilitySet、InputTag 和 RuntimeState API，但不存在受控资源补充或任意 Cooldown 清理 API。

## 架构裁决

采用独立项目插件：

```text
ProjectR runtime module
    ├─ 正式 Combat / GAS / Ability / Save / Travel API
    ├─ 通用日志类别与脱敏工具
    └─ 不依赖 ProjectRDebug

ProjectRDebug runtime plugin
    ├─ UPRDebugSubsystem
    ├─ FPRDebugCommandRegistry
    ├─ 受控命令适配器
    ├─ SProjectRDebugPanel
    └─ F1 Slate input preprocessor
```

插件合同：

- `CanContainContent=false`。
- 模块类型为 Runtime，LoadingPhase 为 Default，仅支持 Win64。
- `.uplugin` 模块配置 `TargetConfigurationDenyList=["Shipping"]`。
- `.uproject` 插件引用再次配置 `TargetConfigurationDenyList=["Shipping"]`。
- `ProjectRDebug` 为读取冻结的 ASC/AttributeSet 状态，私有依赖官方 `GameplayAbilities` 模块，并在 `.uplugin` 中声明对应插件依赖；该依赖不反向进入 ProjectR 主模块，也不改变 Shipping 双重排除。
- 依赖方向只能是 `ProjectRDebug -> ProjectR`；ProjectR 主模块不得依赖、引用或动态加载 ProjectRDebug。
- Development Editor 和 Development Client 启用；Shipping 不参与目标构建。
- 不使用 CheatManager、任意 Console Command、WBP、InputAction 或 MappingContext。

- 结构化命令 Schema、注册表、Provider 生命周期和结构化执行结果。

- v0.1.0-v0.1.4 全量回归、NoUserSaveTouched、NoUEPackages 和 Dirty 门。

`PlayerMessage` 只能使用固定模板和已验证的有限数值，不得包含原始引擎异常或敏感数据。

## `FPRDebugCommandRegistry` 公共合同

```text
FPRDebugProviderHandle RegisterProvider(
    FName ProviderId,
    TConstArrayView<FPRDebugCommandDescriptor> Descriptors,
    FPRDebugCommandHandler Handler)

bool UnregisterProvider(FPRDebugProviderHandle Handle)

TArray<FPRDebugCommandDescriptor> GetCommandDescriptors() const

FPRDebugCommandResult ExecuteCommand(
    const FPRDebugCommandRequest& Request,
    UGameInstance* GameInstance)
```

注册与执行规则：

- ProviderId、CommandId 和 StableName 必须唯一且非空。
- 同一 Provider 实例以完全相同 Descriptor 重复注册时返回原 Handle，不重复插入。
- 同一 ProviderId 但 Descriptor 不同、或不同 Provider 抢占 CommandId/StableName 时必须拒绝并写 `LogProjectRDebug` Error。
- 重复注销返回安全 no-op，不访问失效 Handle。
- Handler 必须为弱 UObject Delegate；失效时返回 `NotAvailable`，不得调用悬空对象。
- RequestId 无效时由 Registry 生成新 GUID；审计日志只把它作为操作关联 ID。
- 未知 CommandId、SchemaVersion 不匹配、缺失/额外/重复参数、错误类型、非有限浮点值或越界参数全部返回 `InvalidParameter`。
- 所有执行必须在 Game Thread；非 Game Thread 返回 `Rejected`，不调度隐式异步任务。
- Registry 不接受自由文本、对象路径、类名、函数名或 Console 文本。
- `GetCommandDescriptors` 按 CommandId 稳定排序，保证 UI 和自动化结果可重复。

## 命令白名单

| CommandId / StableName | 参数 | Availability | 行为 |
|---|---|---|---|
| `StatusSnapshot` / `Debug.Status` | 无 | Available | 显示地图短名、玩家/ASC 存在性、Health/Shield/Energy、Ability/Effect 数量和脱敏 Save 状态 |
| `DamageSelf` / `Combat.DamageSelf` | `Amount`: Float，1–100000，默认 25 | Available | 仅通过 `UPRCombatSubsystem::ApplyDamage` 对当前玩家造成伤害 |
| `ReviveSelf` / `Combat.ReviveSelf` | 无 | Available | 仅通过 `UPRCombatSubsystem::Revive`，Health/Shield fraction 固定为 1 |
| `SaveRuntimeState` / `Save.RuntimeState` | 无 | Available | 仅调用 `UPRSaveSubsystem::GetSaveRuntimeState`，不调用 Create/Load/Save/Delete |
| `ReturnToCombatGym` / `Travel.CombatGym` | 无 | Available | 仅调用 `UPRGameInstance::OpenMap(EPRMapId::CombatGym)` |
| `GrantResource` / `Ability.GrantResource` | `Resource`: EnumChoice={Energy,Permission,Resonance}；`Amount`: Float，0–100000，默认 25 | NotAvailable | 等待正式受控资源 API；不得直接写 Attribute |
| `ClearCooldown` / `Ability.ClearCooldown` | 无 | NotAvailable | 等待正式 Ability/Cooldown API |
| `GenerateDirectorRule` / `Director.GenerateRule` | 无 | NotAvailable | 不创建假 Director 或执行网络请求 |
| `SpawnQTE` / `QTE.Spawn` | 无 | NotAvailable | 不创建假 QTE |
| `CycleCompanionAI` / `Companion.CycleAI` | 无 | NotAvailable | 不创建假 AI 或直接控制 Pawn |
| `JumpToBoss` / `Boss.Jump` | 无 | NotAvailable | 不把现有 BossGym 地图冒充正式 Boss 流程 |

### `Debug.Status`

- 每次执行重新解析当前 World、GameInstance、第一本地玩家、PlayerController、Pawn、PlayerState 和 ASC。
- 只读取 `UPRAttributeSet`、ASC 公共查询和 `UPRSaveSubsystem::GetSaveRuntimeState`。
- 不输出 ProfileId、Spec 完整对象路径、Save Payload 或完整地图 Package 路径。
- 无 Pawn/ASC/SaveSubsystem 时仍返回 `Success` 和明确的存在性字段，不崩溃。

### `Combat.DamageSelf`

- 必须验证 Game Thread、Game World、LocalPlayer、PlayerController、APRPlayerCharacter、APRPlayerState、UPRAbilitySystemComponent 和 UPRCombatSubsystem。
- 构造固定 `FPRDamageRequest`：`SourceId=ProjectR.Debug.DamageSelf`、Target=当前 Character、RawDamage=Amount、非 Critical、无任意 AbilityTag 或 DamageTag。
- 只调用 `UPRCombatSubsystem::ApplyDamage`。
- `Applied -> Success`；Invulnerable/Dead 拒绝映射为 `Rejected`；无效上下文映射为 `InvalidContext` 或 `Failed`。

### `Combat.ReviveSelf`

- 使用与 DamageSelf 相同的完整上下文验证。
- 构造固定 `FPRReviveRequest`：`SourceId=ProjectR.Debug.ReviveSelf`、Target=当前 Character、HealthFraction=1、ShieldFraction=1。
- 只调用 `UPRCombatSubsystem::Revive`。
- Alive 拒绝映射为 `Rejected`；不得自行修改 LifeState 或 Attribute。

### `Save.RuntimeState`

- 只调用 `GetSaveRuntimeState`。
- 不调用 Load、Create、RequestSave 或任何 Storage API。
- 不实现测试清档命令；v0.1.4 的自动化命名空间清理由既有测试基础设施负责，正式账号删除仍属于 v0.4.3。

### `Travel.CombatGym`

- 验证 Game World、`UPRGameInstance` 和 LocalPlayerController。
- 目标固定为 `EPRMapId::CombatGym`；Request 不包含 MapId、路径或字符串参数。
- 只调用现有 `OpenMap` 合同。

## Debug UI 所有权与生命周期

- `UPRDebugSubsystem : UGameInstanceSubsystem` 持有 Registry、内建 Provider、Slate 面板和输入预处理器。
- `SProjectRDebugPanel` 根据 Descriptor 动态生成按钮、Float 输入和有限 EnumChoice，不硬编码 Gameplay 分支。
- 面板显示命令 StableName、描述、Availability、是否改变状态、最近结果和 StatusSnapshot。
- `FPRDebugInputPreprocessor : IInputProcessor` 只处理 F1 KeyDown，不处理重复键事件。
- 打开面板时，仅当 Game/PIE World、GameViewport 和 SceneViewport 存在，且 SceneViewport 具有焦点或鼠标捕获时响应 F1。
- 面板已打开时，除 SceneViewport 外，只允许焦点位于该面板后代控件时用 F1 关闭；其他 Editor 窗口或无关 Slate 控件中的 F1 不响应。
- 不创建 Debug InputAction 或 MappingContext，不修改 `IMC_Player`。
- 打开面板时：只创建一个 Viewport Widget、使用 `GameAndUI`、显示鼠标、不暂停游戏、不修改 Global Time Dilation。
- 关闭面板时：移除准确 Slate Widget，恢复进入面板前的鼠标显示状态，并恢复当前项目基线 `GameOnly`。
- F1 和 Close 按钮均可关闭；重复打开不得产生重复 Widget 或 InputProcessor。
- World cleanup/PreLoadMap 时关闭面板并清除 World/Pawn 弱引用；旅行后不会自动重新打开。
- Pawn 替换时所有命令重新解析当前 Pawn/PlayerState/ASC，不缓存旧对象。
- GameInstance Deinitialize/PIE Stop 时注销 InputProcessor、Provider、Delegate 和 Viewport Widget。
- 当前没有正式 APRHUD，因此不创建 HUD 依赖；未来正式 UI Router 建立后通过新合同接管输入模式恢复。

## 日志 Category 合同

| Category | Default | Compile-time | 用途 |
|---|---|---|---|
| `LogProjectRCore` | Log | All | Framework、地图和生命周期 |
| `LogProjectRInput` | Log | All | 输入语义和映射 |
| `LogProjectRGAS` | Log | All | ASC、AttributeSet、ActorInfo |
| `LogProjectRAbility` | Log | All | Ability 授予、激活、Cost、Cooldown |
| `LogProjectRCombat` | Log | All | Damage、Death、Revive、CombatEvent |
| `LogProjectRSave` | Log | All | Schema、Load、Save、Storage |
| `LogProjectRDebug` | Log | All | 注册、执行、UI、Shipping gate |

- Core/Input/GAS/Ability/Combat/Save 类别由 ProjectR 主模块声明和定义。
- Debug 类别在 ProjectRDebug 插件中声明和定义，因此不进入 Shipping。
- 现有 `LogProjectR` 保留为 Legacy/General；本版本不批量替换现有调用点。
- 新代码必须使用准确类别；后续系统只在其所属版本中增量迁移日志。
- `VeryVerbose` 只用于逐步追踪；`Verbose` 用于详细状态；`Log` 用于正常生命周期；`Display` 用于明确开发操作；`Warning` 用于安全拒绝/回退；`Error` 用于合同或不变量失败。
- 可恢复失败不得使用 Fatal。

## 敏感数据脱敏合同

日志和玩家可见 Debug 结果不得输出：

- Save Payload 或原始 Envelope。
- 完整 ProfileId。
- 用户文件路径、LocalAppData 或平台存储物理路径。
- API Key、认证信息或环境变量值。
- LLM 请求/响应原文。
- 任意未来个人数据、自由文本记忆或账号标识。

`FPRLogSanitizer` 只提供：

```text
FString ToOpaqueGuidToken(const FGuid& Value)
FString RedactedValue()
```

- GUID Token 必须为单向短哈希，不包含 GUID 原始片段。
- Debug 审计日志只包含 RequestId、StableName、ResultCode、`bChangesRuntimeState` 和脱敏上下文。
- Save RuntimeState 的玩家可见结果完全省略 ProfileId，不以哈希代替显示。

## Shipping 隔离合同

- `ProjectRDebug.uplugin` 的 Runtime Module 使用 `TargetConfigurationDenyList=["Shipping"]`。
- `ProjectR.uproject` 的插件引用同样使用 `TargetConfigurationDenyList=["Shipping"]`。
- 插件 `CanContainContent=false`，没有任何 Debug UE Package。
- ProjectR 主模块不得在 Build.cs、Source、Config 或软路径中依赖 ProjectRDebug。
- `UPRDeveloperSettings::bEnableDebugFeatures` 的 C++ 默认值：Editor/Development 为 true，Shipping 为 false。
- 插件缺席是第一隔离层；`UE_BUILD_SHIPPING`/运行时 Gate 只能作为防御层，不能替代模块排除。
- `WITH_EDITOR` 只用于 Editor 测试或 MCP；不能用于排除 Development Client 所需功能。
- Shipping 收据、Stage Manifest、Binary、Plugin 目录和启动行为必须分别验证不存在 ProjectRDebug。
- 不能以隐藏按钮、Collapsed Widget 或未绑定输入冒充 Shipping 隔离。

## MCP / Editor Operation Manifest

## 回滚和停止条件

以下任一情况必须停止并只反向撤销本版本准确 diff：

- ProjectR 主模块必须依赖或动态加载 ProjectRDebug。
- Shipping 收据、Stage、Binary、Plugin 或运行时包含 ProjectRDebug。
- 必须修改 Config、正式输入、地图或 UE Package 才能完成。
- 必须新增任意 Console、反射、Spawn、文件、网络、脚本或资产执行入口。
- Debug 命令绕过正式 Combat、Save 或 Travel API。
- Save 查询产生写入、删除或生产 Slot 访问。
- 注册、旅行、Pawn 替换或 PIE 循环产生重复 Handler、Delegate、InputProcessor 或 Widget。
- F1 在 Shipping 中有响应。
- Dirty Packages 无法归零。
- BuildEditor、Development Package、Shipping Package、required AutomationReport 或既有回归失败。
- GameplayTag、正式输入、Blueprint API、Save Schema 或既有 UE Package 发生变化。

回滚禁止使用 `git reset --hard`、`git clean`、Save All、Fix Redirectors、批量删除或修改用户 Save。只允许精确撤销本版本新增/修改的文本文件和插件目录。
