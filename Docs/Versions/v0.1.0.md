# v0.1.0 Enhanced Input、2.5D 移动与侧视相机

**状态**：Completed

**直接上游**：v0.0.5

**直接下游**：v0.1.1

**合同消费者**：v0.1.3、v0.3.2

## 目标

建立正式、数据驱动的玩家输入层和可验证的 2.5D 基础移动：

- Codex 通过 Unreal MCP 创建正式 InputAction、InputMappingContext 和 InputConfig DataAsset。
- 键盘、鼠标与手柄输入转换为稳定的 `Input.*` 语义意图。
- 玩家只能在 X/Z 作战平面内移动和跳跃。
- 角色根据 X 方向移动朝向左或右，并在停止移动后保持最后有效朝向。
- 跳跃中收到反方向输入时，角色必须立即切换实际 X 移动方向，而不只是改变 Mesh 朝向。
- 保持 v0.0.3 冻结的固定侧视相机配置。
- 为 v0.1.3 GAS 输入路由和 v0.3.2 QTE 提供增量扩展点，但不实现未来业务逻辑。

## 冻结合同与未来扩展点

- 正式输入资产根路径冻结为 `/Game/ProjectR/Input/`。
- `Input.*` 只表示玩家语义意图，不表示具体 Ability、QTE 或 UI 类。
- Mapping Context 由 PlayerController/LocalPlayer 生命周期管理。
- PlayerCharacter 负责移动、跳跃和语义输入钩子。
- 正式地图路径、`OpenMap` 合同、GameplayTag 名称和现有 Blueprint 父类不得改变。
- 默认键位是初始配置，不是永久不可变的玩家设置；后续重绑定系统可以替换映射而不改变语义 Tag。
- `BP_PlayerCharacter` 的相机相对位置 `(0,600,100)`、相对旋转 `(0,-90,0)` 与 FOV 60 保持兼容。

| 语义 | InputAction 类型 | 键盘/鼠标 | 手柄 |
|---|---|---|---|
| Move | Axis1D | A = -1，D = +1 | Left Stick X |
| Jump | Boolean | Space | Face Button Bottom |
| Attack | Boolean | Left Mouse Button | Face Button Left |
| Dodge | Boolean | Left Shift | Face Button Right |

| Execute | Boolean | F | Right Shoulder |

- Tag 和 Action 均不得重复；空引用、重复 Tag、重复 Action 或错误 ValueType 必须验证失败。

## 2.5D 行为合同

- 作战平面法线固定为 Y 轴。
- 角色生成后以当前 Y 坐标作为平面原点并吸附到该平面。
- Move 只向世界 X 轴提交移动输入。
- Falling 状态下，如 Move 方向与当前 X 速度相反，立即将 X 速度按原绝对值切换到输入方向；Z 速度、Y 平面、地面参数、最大速度与跳跃参数不变。
- Jump 只由 CharacterMovement 改变 Z。
- PIE 中移动和跳跃期间 Y 偏差不得超过 `0.1 cm`。
- `bUseControllerRotationYaw = false`。
- CharacterMovement 根据移动方向朝向 `+X` 或 `-X`。
- 停止移动后保持最后有效朝向。
- 不启用自由 Look。
- 保持现有侧视相机相对位置、旋转和 FOV，不由本版本重新调参。

## MCP Operation Manifest

所有 Editor 写入必须串行，按以下顺序执行：

1. **只读能力与碰撞审计**
   - 描述 DataAsset、Object、Blueprint、Asset、PIE、Log 和 Automation 相关 Toolset。
   - 对全部目标 Package 执行 exists/can-edit 检查；任一目标已存在即停止，不覆盖。
   - 验证模板、Variant 和 MCPTest 输入资产只读且不进入正式引用链。
2. **C++ 前置门**
   - 完成 BuildEditor。
   - 正常重启 Editor，使新反射类型加载后再创建 DataAsset。
3. **正式资产创建**
   - 优先使用官方 `DataAssetTools` 创建 InputAction、InputMappingContext 和 InputConfig。
   - 使用官方 `ObjectTools` 设置 InputAction ValueType、IMC `DefaultKeyMappings`、A 键负值修饰以及 InputConfig 的 MappingContext、Priority 和 Tag/Action 数组。
4. **官方工具能力缺口门**
   - 若官方工具不能写入嵌套 Mapping，保存工具名、参数、返回值与错误日志。
   - 不把机械资产操作转交用户，也不部分保存正式资产。
   - 停止并提出最小 ProjectR editor-only Toolset 方案，等待单独批准。
5. **Blueprint 配置**
   - 设置 `BP_PlayerController` CDO 的 InputConfig 引用。
   - 编译 `BP_PlayerController` 和 `BP_PlayerCharacter`；warnings-as-errors。
6. **精确保存与 Dirty 门**
   - 只保存 Manifest 中实际创建或修改的非空 Package 集。
   - 查询全量意外 Dirty Packages，要求为 0。
7. **重启回载**
   - 正常关闭并重启 Editor。
   - 重新读取资产类、ValueType、13 条 Mapping、六组 Tag/Action、Blueprint 父类/CDO 引用及 PlayerCharacter 相机配置。
8. **PIE 与报告**
   - 打开 `L_CombatGym`，运行 PIE、采集日志和截图、停止 PIE。
   - 生成 AutomationReport，分别记录 Build、资产创建、Blueprint 编译、精确保存、Dirty、重启回载、引用、自动输入和 PIE 状态。

## 人工判断与准确步骤

Codex 负责完成全部机械操作和客观验证。用户只进行无法由自动化替代的主观手感确认：

1. Codex 完成全部自动验证并在 Editor 中打开 `L_CombatGym`。
2. 用户开始 PIE。
3. 按住 A、D，确认实际移动方向立即改变，Mesh 约 0.12 秒平滑转向，停止后保持最后朝向。
4. 移动中按 Space，确认跳跃没有可感知的前后/Y 轴漂移。
5. 朝一个方向跳起后在空中按相反方向，确认实际移动立即反向；快速交替 A/D 并连续跳跃，确认 Mesh 从当前角度连续转向、相机保持固定侧视且无抖动。
6. 判断当前速度、跳跃高度和相机构图是否达到“可继续开发”的基础水平。
7. 停止 PIE，并把主观判断记录为 PASS 或 FAIL；不在本版本顺便实现攻击、闪避或技能。
