# ProjectR v0.2.4 第一轮战斗手感和平衡

## 版本关系与目标

- 直接上游为 v0.2.3，直接下游为 v0.3.0；未来消费者为 v0.3.1、v0.3.2、v0.4.0、v0.4.2、v0.4.4、v0.7.x、v0.8.x、v0.9.1 与 v1.0.1。
- 本版本是单一完整路线版本：不得拆为 A–E、子版本、阶段版本、检查点版本或中间功能提交。唯一正式功能实现提交为 `v0.2.4 First combat feel and balance pass`。
- 目标是在不改变既有公共玩法合同的前提下形成首轮功能验证数值基线。固定 CombatGym 五次与 Auditor Boss 三次人工运行仍须记录；用户已明确将原 60-120 秒与 180-300 秒时长范围降级为后续关卡节奏优化观察项，而非本版本数值调参或完成门。
- 玩家必须存在通过正式 `UPRCombatSubsystem` 造成的真实死亡风险，但不得用不可避免伤害、强制扣血或测试捷径伪造风险。
- 每个路线列出的候选字段最终均标记为 `Changed`、`Retained` 或 `Not Applicable`；不得为了制造 diff 盲目修改合理数值。

## 冻结合同与 Non-goals

- 原始基线为 126 个显式 GameplayTag、13 个根命名空间、60 个 checked getter。经用户明确批准的本版本合同修订只追加 `Skill.BasicAttack` 与其 checked getter：现为 127/13/61；既有 126 个 Tag、名称、注释与语义不变，无 Redirect。
- 冻结 X/Z 作战面、Y 轴约束、固定侧视相机、PlayerState Owner/Character Avatar/ASC/AttributeSet 关系、六技能 Tag/InputTag/AbilitySet 顺序与路径、CombatSubsystem 唯一伤害链、Enemy Registry 白名单、四种 Enemy 身份、Auditor 阶段/P2 本地规则/P3 PredictionShield/一次 Completion 与运行时 1 碎片、以及 Combat HUD 的只读事件消费者关系。
- 不实现 Companion、关系、QTE、Director、LLM、Room、Encounter、Reward、Roguelike、Meta、账号、墓园、章节、Save Schema 或用户存档业务。
- 不创建敌人、Boss 攻击、GameplayEffect、地图、HUD、Widget、Input 资产、Balance DataAsset、Balance Subsystem、遥测或 Analytics。经用户明确批准，本版本唯一玩法例外为正式 `Skill.BasicAttack` 的 DataAsset/GA 二个 Package；它不属于六项 P0 PlayerSkill，不增加 InputAction、Cost/Cooldown GE、表现资产或第二套战斗系统。
- 不修改 Combat、AttributeSet、ASC、六项 GameplayAbility、Enemy Brain/StateTree、BossSubsystem、Combat HUD、Save、Debug、Input、`.uproject` 或正式地图。经用户明确批准的精确例外为 `Config/DefaultGameplayTags.ini` 追加 `Skill.BasicAttack`、`UPRTagLibrary` getter、`DA_DefaultAbilitySet` 第七项和对应 BasicAttack C++/测试；若需要任何其他禁止路径，必须停止并提出新的合同修订。

1. 先对全部候选资产进行只读回读，输出 Git ignored 的 `Saved/BalanceReports/` Baseline Snapshot；记录 Package、字段、数值、引用、LFS/OID 或可复核哈希与测试断言位置。
2. `Docs/BalanceNotes.md` 仅在实际实施期创建，记录 Baseline、Final、Delta、理由、自动化证据、5+3 人工记录、异常值、回滚值和后续观察项；它不是运行时数值来源。
3. 只有有证据的字段可修改。候选权威来源为：
   - 玩家：`/Game/ProjectR/Effects/GE_DefaultAttributes` 的 MoveSpeed；`/Game/ProjectR/Blueprints/Player/BP_PlayerCharacter` 的既有 CharacterMovement CDO；`DA_Skill_AfterimageDodge` 的位移/窗口。
   - 技能：六个 `/Game/ProjectR/Abilities/Skills/DA_Skill_*`，以及各自现有 Cost/Cooldown GameplayEffect。
   - Enemy：四个 `DA_Enemy_*` Prototype、四个 `DA_EnemyAttack_*` 与四个 `GE_EnemyAttack_*_Cooldown`。
   - Boss：`DA_Boss_Auditor`、三项 `DA_EnemyAttack_Auditor*`、三项 `GE_EnemyAttack_Auditor*_Cooldown` 与 `GE_Boss_Auditor_PredictionShield`。
4. `GE_Enemy_DefaultAttributes`、`DA_DefaultAbilitySet`、所有 GA、Enemy AbilitySet、Registry、StateTree、Enemy/Boss Blueprint、HUD/Widget/GameMode、Input/IMC/InputConfig、`L_CombatGym`、`L_BossGym`、MCPTest、Template 与 Variant 均为 Forbidden。

### 已批准 BasicAttack 与死亡碰撞修订

- `DA_DefaultAbilitySet` 的前六项和顺序保持冻结；只追加第七项 `/Game/ProjectR/Abilities/BasicAttack/GA_BasicAttack`，其 SourceObject 为 `/Game/ProjectR/Abilities/BasicAttack/DA_BasicAttack`、`Input.Attack`、Level=1、Initialization=true、无 GrantedSpecTags。
- BasicAttack 固定为 `Skill.BasicAttack`、`OnInputTriggered`、ForwardSweep 150 cm/45 cm、`10 + 1.0 * AttackPower`、startup/active/recovery 为 `0.08/0.06/0.22 s`、零 Energy、零 Cooldown。它只经 `UPRCombatSubsystem::ApplyDamage`，不改变六项 P0 PlayerSkill、InputAction、表现预加载或 HUD 的六个技能槽。
- `APREnemyCharacter` 死亡后仍保留 Actor 供未来 Revive，但 Capsule 对 `ECC_Pawn` 立即改为 Ignore；Revive 只恢复该 Actor 原始 Pawn 响应。此规则不改变 Alive Enemy 的碰撞、Combat、AttributeSet、Registry 或 Spawn 行为。
- 此修订准确创建 `/Game/ProjectR/Abilities/BasicAttack/DA_BasicAttack` 与 `/Game/ProjectR/Abilities/BasicAttack/GA_BasicAttack`，并修改 `/Game/ProjectR/Abilities/DA_DefaultAbilitySet`；不保存地图或其他既有资产。
5. 产品运行时 C++ 默认不修改；允许同步更新的仅为实际变更数值的精确自动化期望。若值只存在于硬编码 C++，先记录准确文件、字段、消费者、兼容影响和数据化替代方案，再请求合同修订。

## 固定基准与人工边界

- 普通战斗固定为 `L_CombatGym`：通过既有固定白名单 editor-only 入口生成 2 Melee、1 Ranged、1 Shield；固定 X/Z 生成点；不含 Elite、不保存地图、不修改 Registry。Elite 仅通过独立自动回归验证。
- Boss 固定为 `L_BossGym` 的 Auditor，保留 P1/P2/P3、PredictionShield 与 Completion；不得用 Debug 跳阶段、自动秒杀、持久化碎片或删除 Profile/Slot。
- Codex 负责数值审计、Baseline 导出、自动指标、测试、MCP 修改、回读、精确保存、重启、PIE、日志和最终报告。
- 所有客观 required 门 PASS 后，用户只负责连续完成 5 次固定 CombatGym 与 3 次 BossGym，并给出移动/跳跃/闪避/攻击节奏、Boss 前摇/间隔/阶段节奏的主观结论。Codex 记录时长、资源、Damage Taken、受击、技能使用、Energy/Cooldown 失败、击杀/阶段时间、PredictionShield 与异常值。

## MCP Operation Manifest

1. 串行确认唯一 Editor、`PIE=false`、初始 Dirty=0，并审计 Asset/Object/DataAsset/Blueprint/GameplayEffect、PIE、Log、Automation 与现有 ProjectR Toolset。
2. 对每个候选 Package 检查存在性、类型、可编辑性、当前字段、引用、Dirty 和碰撞；导出不写资产的 Baseline。
3. 运行基线 Build、Automation 与固定 PIE，形成有证据的精确调参表。
4. 仅通过官方 Toolset 原位修改已审计字段；若嵌套字段能力确有缺口，先记录 Tool/参数/返回/日志，再允许最小固定、无参数、editor-only `PRBalanceAutomationToolset`。该工具只运行固定 CombatGym/BossGym、白名单 Enemy、只读事件和 `Saved/BalanceReports/`，不得接受任意路径、类、Tag、命令、代码、Save Slot 或自由文本。
5. 完整回读；任一类型、字段、引用或顺序不符即停止且不保存。warnings-as-errors 编译受影响 Blueprint/GameplayEffect；只保存实际 Dirty 的精确 Manifest；检查意外 Dirty=0；正常重启并再次回读。
6. 运行自动基准、5+3 人工 Runbook；StopPIE 后确认 Actor、Timer、Delegate、Widget、Effect、Spec 与 Dirty 清理；生成 Final Snapshot、BalanceNotes 与 AutomationReport。
