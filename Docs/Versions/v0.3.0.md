# ProjectR v0.3.0 Companion 数据、关系与主同步结构

- 直接上游为 v0.2.4，直接下游为 v0.3.1；未来消费者为 v0.3.2、v0.3.3、v0.3.4、v0.4.3 与 v0.5.1。

- 建立律衡（Axiom）、燧火（Kindle）、鸦零（Null）三份正式 DataAsset、固定 Registry、关系快照/变化事件、每轮唯一主同步选择和版本化 Save Schema 2。

## 范围与 Non-goals

本版本范围：

- 新增 Core 关系值类型，三个 Companion 的稳定 PrimaryDataAsset 与固定 Registry。
- 新增 `UGameInstanceSubsystem`，提供唯一/幂等主同步选择、后台派生、只读关系查询及受控关系 Delta。
- Save Schema 1→2：只持久化三名 Companion 的长期关系；新 Profile 和旧 Schema 1 均获得确定性默认关系。
- 执行原生自动化、Build、资产 Data Validation、精确保存、Dirty=0、重启回载、固定 PIE selection/travel smoke 与所有相关回归。

Non-goals：

- 不实现 Companion Actor、跟随、战斗、AIController、BehaviorTree、StateTree、支援 Ability、压血不击杀、QTE、对话、动态台词、濒死保护、关系 UI 或肖像 Widget。
- 不持久化主同步选择；它是每轮运行时状态，在 Profile Create/Load、PIE/Engine 生命周期结束时清空，在普通地图旅行后保持。
- 不新增 GameplayTag、地图、GameMode、Widget、Input、Ability、GameplayEffect、Save Slot API 或用户可写的通用 Profile API。
- 不读写真实用户存档；自动化只使用唯一 GUID 白名单 A/B 槽。

## 冻结公共合同

### Relationship 类型

`FPRRelationshipState` 固定字段：`Trust`、`Affection`、`Evaluation`、`Overload`。四项均为 `int32`，范围 `[0,100]`；默认值依次为 `50`、`50`、`50`、`0`。

`FPRCompanionRelationshipRecord` 固定字段：`CompanionId`、`State`。Canonical 顺序固定为 Axiom、Kindle、Null，不接受缺项、重复、未知 ID 或乱序记录。

`FPRRelationshipDelta` 固定字段：`CompanionId`、四项整数 Delta、`SourceId`。`SourceId` 必须非 `NAME_None`；应用后逐字段 Clamp。Delta 不含 UObject、Spec、EffectHandle、Timer、Delegate、网络或 Save Slot。

`FPRRelationshipChangedEvent` 固定字段：`CompanionId`、`PreviousState`、`CurrentState`、`AppliedDelta`、`WorldTimeSeconds`。只在实际值变化后广播一次。

### Companion Data 与 Registry

`UPRCompanionDataAsset : UPrimaryDataAsset` 固定包含：`CompanionId`、`DisplayName`、`RoleLabel`、`PersonaSummary`、`AccentColor`。不含未来 Ability、AI 类、QTE、台词、对话图、关系数值、奖励、存档或软对象引用。

`UPRCompanionRegistryDataAsset` 固定包含三项软引用，顺序为 Axiom、Kindle、Null；Data Validation 必须断言路径、PrimaryAssetId、Tag、显示文本与唯一性。固定路径为 `/Game/ProjectR/Companions/DA_CompanionRegistry`；不使用 Config、自由路径、字符串或任意 UObject 参数。

### Subsystem API 与不变量

`UPRCompanionSubsystem : UGameInstanceSubsystem` 为选择和关系的唯一 owner，受控 C++ API 固定为：

```text
EPRPrimaryCompanionSelectionResult SelectPrimaryCompanion(FGameplayTag CompanionId)
FPRCompanionSyncState GetSyncState() const
bool GetRelationshipSnapshot(FGameplayTag CompanionId, FPRCompanionRelationshipRecord& OutRecord) const
void GetAllRelationshipSnapshots(TArray<FPRCompanionRelationshipRecord>& OutRecords) const
EPRRelationshipApplyResult ApplyRelationshipDelta(const FPRRelationshipDelta& Delta)
FPRRelationshipChangedNative& OnRelationshipChanged()
FPRPrimaryCompanionSyncChangedNative& OnPrimarySyncChanged()
```

- registry 未 ready、未知 Tag 或非法 Delta 必须拒绝且不改状态；重复选择同一 Primary 返回幂等结果，不重新广播。
- 每次有效选择恰有 1 名 Primary、2 名 Background；Background 为 Registry 顺序中其余两名。
- Subsystem 只订阅 Save 的成功 Create/Load 事件，以加载 canonical relationship snapshots；绑定/解绑必须幂等，不使用 Tick。
- `ApplyRelationshipDelta` 仅 stage 已加载 Profile 的关系分区并标记需要显式 Save；不会发起 I/O。关系由 CompanionSubsystem 负责，SaveSubsystem 不依赖 Companion 类型。

## Save Schema 2 与迁移

- `FPRProfileSaveData` 在既有 `ProfileId` 后追加 `CompanionRelationships`；当前 SchemaVersion 由 1 升至 2，MinimumMigratableVersion 保持 1。
- 新建 Profile 立即创建 canonical 默认记录；首次落盘仍遵循既有 Revision 1 规则。
- 生产迁移仅注册精确 `1 → 2`：在 SaveMigration 的临时 duplicate 上填入默认 records 并令 Schema=2；原对象、Revision、ProfileId 和 A/B selection 规则不变。
- Schema 0 仍为 MissingSchemaVersion；未来 Schema/Envelope 一律拒绝，不能回退或覆盖；损坏备用代只按既有恢复合同处理。
- `UPRSaveSubsystem` 只可新增窄 C++ snapshot/stage API，严格校验 canonical records；不得暴露任意 Profile、Schema、路径、Slot、删除或磁盘写入入口。
- Migration 成功只 `bNeedsResave=true`，不在 Load 中自动写盘；任何完整性失败都保留迁移前已发布对象。

## MCP Operation Manifest

1. 确认唯一 Editor、PIE=false、初始 Dirty=0，并审计官方 DataAsset/Object/Blueprint/PIE Toolset。
2. Build C++ 后正常重启，确认所有反射类型；检查四个 Create path 不存在且可编辑。
3. 逐项创建、配置与回读三份 Companion Asset 和 Registry；断言三项顺序/Tag/文本/颜色及无未来引用。
4. Data Validation，精确保存仅上述实际 Dirty Package，检查意外 Dirty=0。
5. 正常重启后回读 PrimaryAssetId、Registry 顺序和引用；不加载或保存地图。
6. 运行固定无参 PIE smoke：选择 Axiom、验证 1+2 派生、地图旅行后保持、Profile load/create 后清空；不产生 Save I/O。
7. StopPIE 后检查委托、runtime state、生成 Actor、Timer 和 Dirty 清理，生成 AutomationReport。
8. 官方工具无法完整配置/回读时，先保存 Tool/参数/返回/日志证据，再建立最小固定 editor-only Toolset；禁止将机械创建交给用户。

## 人工步骤、停止条件与回滚

- 本版本没有必需主观人工验收；用户不创建或配置任何资产。若最终 PIE 表现有可见 UI/Actor 之外的客观失败，由 Codex 修复与复验。
- 若必须新增 Tag/Config/地图/未来玩法资产、自动保存/通用 Slot API、修改 Combat/Input/Ability/Enemy/Boss/HUD 合同、允许未知 Companion 或无法保证 Save 迁移/精确保存/Dirty/重启/PIE 清理，立即停止并提出合同修订。
- 回滚仅反向撤销本版本准确 C++/文档/Manifest 引用；已保存 Package 删除须逐项批准。禁止 hard reset、git clean、Save All、Resave All、Fix Redirectors、批量删除和普通文件 I/O 修改 UE Package。

## Future Compatibility Review

- v0.3.1 Actor/跟随/战斗支援只读取稳定 Data/relationship/sync snapshot，不能直接改 Save 或绕过 CombatSubsystem。
- v0.3.2 QTE 只通过受控 `FPRRelationshipDelta` 产生关系结果；不反向控制实体、Save 或 Combat。
- v0.3.3 对话/动态台词只消费 persona/relationship event，不保存原始文本或网络等待状态。
- v0.3.4 濒死保护与分歧缓存以 Primary/relationship snapshot 为输入，救援 token 保持 run-local。
- v0.4.3/4.4/5.1 的 Account/Run/Meta/Task 分区必须通过递增 Schema 迁移加入；不能替换 ProfileId、Schema 2 relationship 语义或 A/B envelope。
